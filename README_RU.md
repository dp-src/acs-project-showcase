## Подход к разработке

Проект полностью реализован **с помощью ИИ-агентов на основе детальных спецификаций**.

Каждый модуль, endpoint, адаптер, миграция БД и интерфейс были сгенерированы ИИ по текстовым описаниям. Роль создателя сводилась к формированию требований, описанию желаемого поведения и верификации результата на реальном оборудовании — без написания кода вручную.

## Обзор

**S-ACCESS** — автономная локальная (self-hosted) система контроля и управления доступом (СКУД) для среды Windows. Она служит единым бэкендом для биометрических терминалов, считывателей карт и турникетов, объединяя обработку событий в реальном времени, учёт рабочего времени, управление пользователями и синхронизацию устройств в рамках одного сервера.

**Философия:** «Настроил и забыл» — система спроектирована так, чтобы не требовать обслуживания после внедрения.

**Целевая аудитория:** Средние и крупные организации (500+ сотрудников) с несколькими точками входа/выхода.

## Технологический стек

| Уровень | Технология |
|---|---|
| **Платформа** | Node.js 18+ |
| **Веб-сервер** | Express.js 4.18 |
| **База данных** | SQLite 3 (WAL mode, crash-safe) |
| **Фронтенд** | Vanilla JavaScript, HTML5, CSS3 — без фреймворков |
| **Хеширование** | bcrypt (10 раундов) |
| **Шифрование** | AES-256-CBC (уникальный IV для каждой записи) |
| **Протоколы устройств** | HTTP Digest (Hikvision) · TCP DVRIP бинарный (Dahua) |
| **Логирование** | Ротация файлов (лимит 50 МБ, хранение 5 архивов) |
| **Сборка** | `pkg` — компиляция в единый `.exe` для Windows x64 |
| **Уведомления** | node-telegram-bot-api |
| **Экспорт** | ExcelJS |
| **Парсинг** | xml2js (XML-поток событий Hikvision) |

```
┌─────────────────────── S-ACCESS СЕРВЕР ──────────────────────────┐
│                                                                  │
│  Express HTTP API  ←→  Аутентификация  ←→  SQLite (WAL)          │
│         │                                     │                  │
│   REST-маршруты                        Индексы в памяти          │
│   (users/devices/                      uidIndex / cardIndex      │
│    logs/stats/system)                  Поиск O(1)                │
│         │                                                        │
│   Сервисный слой                                                 │
│   ├── eventHandler      (разрешение ENTRY/EXIT + сессии)         │
│   ├── deviceManager     (жизненный цикл, heartbeat, оркестрация) │
│   ├── syncService       (распространение данных пользователей)   │
│   ├── eventSyncService  (историческое восстановление с устройств)│
│   ├── backupService     (плановые и ручные резервные копии)      │
│   ├── maintenanceService(монитор диска, VACUUM, очистка фото)    │

│   └── shiftReportService(уведомления в Telegram)                 │
│         │                                                        │
│   Слой адаптеров   [Паттерн Адаптер — абстракция протоколов]     │
│   ├── HikvisionAdapter  → HTTP ISAPI / Digest Auth               │
│   └── DahuaAdapter      → TCP DVRIP / Двоичный протокол          │
│                                                                  │
└──────────────┬─────────────────────────┬─────────────────────────┘
               │                         │
   ┌───────────▼──────────┐  ┌───────────▼──────────┐
   │  Терминалы Hikvision │  │   Турникеты Dahua    │
   │  DS-K1T343M          │  │   ASG-ASI2101        │
   │  Лицо + Карта        │  │   Карта              │
   │  Порт 80 · REST/XML  │  │   Порт 37777 · TCP   │
   └──────────────────────┘  └──────────────────────┘
```

### Ключевые архитектурные решения

- **Без фронтенд-фреймворка** — Vanilla JS обеспечивает многолетнюю совместимость без устаревания зависимостей
- **Паттерн Адаптер** — новые производители устройств могут быть добавлены без изменения бизнес-логики
- **Шина событий (pub/sub)** — несвязанный поток событий между мониторами устройств и обработчиком сессий
- **Зеркалирование конфигурации** — настройки хранятся как в SQLite, так и в JSON-файле; переживает повреждение БД
- **Привязка лицензии к аппаратуре** — ключ производится из UUID материнской платы и ID процессора; не может быть клонирован

## Интеграции с устройствами

### Hikvision DS-K1T343M — Терминал распознавания лиц

- **Протокол:** HTTP REST (ISAPI) с Digest-аутентификацией
- **Возможности:** Считыватель карт, распознавание лиц
- **Доставка событий:** Long-polling чанкированный XML-поток (`/ISAPI/Event/notification/alertStream`)
- **Формат карты:** 10-значное десятичное число с ведущими нулями

### Dahua ASG-ASI2101 — Контроллер турникета

- **Протокол:** TCP DVRIP — проприетарный двоичный протокол (32-байтный заголовок + JSON-нагрузка)
- **Возможности:** Считыватель карт
- **Доставка событий:** TCP-подписка с постоянной сессией
- **Формат карты:** Шестнадцатеричный (например, `060BE834`)

### Автообнаружение

Сервер сканирует локальную сеть и автоматически определяет производителей устройств:
- TCP SYN на порт `37777` → Dahua
- HTTP-запрос на порт `80` / `8000` → Hikvision

UID карт нормализуются между форматами прозрачно как при приёме, так и при синхронизации.

## Безопасность

| Угроза | Реализация |
|---|---|
| Пароль администратора | Хеширование bcrypt, хранение в открытом виде исключено |
| Пароли устройств | Шифрование AES-256-CBC (data at rest), уникальный IV для каждой записи |
| Биометрические фото | Шифрование AES-256-CBC на диске (`{uid}.enc`), расшифровка только в момент синхронизации |
| Токены сессий | UUID v4, валидация при каждом запросе |
| Валидация ввода | Санитизация и строгая типизация на всех точках входа API |
| Защита от Timing Attacks | Использование `crypto.timingSafeEqual` для проверки лицензионных ключей |
| Чистые логи | Пароли, токены и ключи никогда не попадают в лог-файлы |
| Привязка к железу | Лицензионный ключ = SHA-256(HWID + соль); перенос невозможен |

## Модель лицензирования

S-ACCESS распространяется как скомпилированный Windows `.exe` без внешних зависимостей среды выполнения.

- **Тип лицензии:** Проприетарная, привязанная к аппаратуре — одна лицензия на серверную машину
- **Активация (BETA):** Каждый ключ производится из UUID материнской платы и ID процессора сервера; он не может быть перенесён на другую машину
- **Модель развёртывания:** Устанавливается на объекте клиента системным интегратором
- **Без облачной зависимости:** Работает полностью офлайн после активации
